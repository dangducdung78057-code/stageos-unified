# 构建一个 2.5D 多人拖拽交互式换装应用（v2 · 渲染与开源精灵增补版）

> v2 变更说明：新增「渲染管线设计规范」「开源人物精灵资源方案」两大章节；学习目标、仓库规划、验收标准、风险清单同步更新。原有七步学习者旅程保持不变。

## Summary

本练习引导学习者从零构建一个基于 HTML5 Canvas 的 2.5D 等距视角多人换装室。学习者将逐步实现等距场景渲染、精灵层级排序、拖拽换装交互、WebSocket 实时多人同步以及装备槽位状态管理。最终产物是一个可在浏览器中运行的多人协作换装应用，多位玩家可同时在线拖拽服饰部件为角色换装，所有操作实时同步给其他玩家。

## Learning objectives

完成本练习后，学习者将能够：

- 实现等距（isometric）坐标系统，将 2D 屏幕坐标与等距世界坐标相互转换
- 设计并实现基于深度排序的精灵层级渲染管线，解决 2.5D 场景中的遮挡问题
- 【v2 新增】实现多层精灵合成渲染：按固定层级栈合成角色与装备，统一锚点对齐
- 【v2 新增】处理 HiDPI 适配与离屏 Canvas 高清导出，掌握图层重着色（tint）技术
- 实现基于 HTML5 Drag and Drop API 与 Canvas 坐标检测的混合拖拽交互系统
- 搭建 WebSocket 服务端，实现多客户端实时状态广播与同步
- 设计装备槽位数据模型，管理角色部件的挂载、替换与卸下
- 【v2 新增】评估开源精灵资源的授权协议（CC0 / CC-BY / CC-BY-SA / GPL），为项目做出合规的素材选型
- 编写 GitHub Actions 工作流，实现自动化步骤验证与学习者反馈

## 【v2 新增】渲染管线设计规范

### 1. 渲染层级栈（Layer Stack）

每一帧按以下固定顺序绘制，**实体间**用深度排序（`y + x`），**单个角色内部**用固定层级序号，两套排序互不干扰：

| 层级 | 内容 | 说明 |
| --- | --- | --- |
| L0 | 地面 tile 层 | 等距菱形地板，按行从远到近绘制 |
| L1 | 地面贴片层 | 角色脚下的椭圆投影（blob shadow）、站位高亮框 |
| L2 | 实体层（深度排序） | 所有角色/道具按 `y + x` 排序后逐个绘制；单个角色内部按下表 L2a–L2f 固定叠层 |
| L2a | 下装（bottom） | 最贴近角色底模 |
| L2b | 鞋子（shoes） | 覆盖下装裤脚 |
| L2c | 角色底模（body） | 皮肤层，被服装覆盖的区域实际被遮挡 |
| L2d | 上装（top） | 覆盖底模躯干与下装腰部 |
| L2e | 头部/发型（head） | 含头饰基底 |
| L2f | 饰品（accessory） | 最顶层，如帽子、手持物 |
| L3 | UI 叠加层 | 玩家名称标签、拖拽悬停高亮、连接状态指示 |

> 实现要点：L2 的实体排序键与 L2a–L2f 的层内序号合并为一个复合键 `depth = (y + x) * 10 + layerIndex`，一次排序完成全部绘制，避免两轮遍历。

### 2. 锚点规范（Anchor Convention）

所有精灵（底模与全部装备件）必须遵守同一锚点约定，这是"换装零偏移"的前提：

- **角色与装备**：锚点 = 脚底中心，即归一化坐标 `(0.5, 1.0)`。绘制时 `drawImage(sprite, worldX - w/2, worldY - h)`。
- **地面 tile**：锚点 = 菱形中心 `(0.5, 0.5)`。
- **所有槽位精灵统一画布尺寸**（如 128×192），空区域透明。这样换装 = 直接替换对应层纹理，**不需要为每件装备配置 offset**——偏移量在设计素材时一次性对齐，而不是在代码里逐件补偿。

### 3. 换装过渡与反馈

- **过渡动画**：旧装备 alpha 从 1→0、新装备 0→1 交叉淡入，时长 150ms；或新装备 1.15→1.0 缩放弹跳（pop），二选一，全应用统一。
- **拖拽悬停反馈**：拖拽物进入角色命中区时，L1 层绘制槽位高亮框 + 角色外发光描边（shadowBlur）。
- **失败反馈**：拖到无效区域时装备图标回弹动画（CSS transform 过渡即可，无需 Canvas 参与）。

### 4. HiDPI 适配

Canvas 在高分屏（devicePixelRatio > 1）下会模糊且坐标偏移，模板中必须预设：

```
canvas.width  = cssWidth  * devicePixelRatio
canvas.height = cssHeight * devicePixelRatio
ctx.setTransform(devicePixelRatio, 0, 0, devicePixelRatio, 0, 0)
```

此后**全部逻辑坐标使用 CSS 像素**，禁止在业务代码中直接读写 `canvas.width`。窗口 resize 时重新执行上述设置。

### 5. 离屏渲染与高清导出

为满足"导出当前画面为高清 PNG"的需求（StageOS 队形图导出同源能力）：

1. 创建离屏 Canvas，尺寸 = 场景逻辑尺寸 × 导出倍率（2x/3x）；
2. 用完整渲染管线向离屏上下文重绘一帧（渲染函数必须上下文无关，可传入任意 `CanvasRenderingContext2D`）；
3. `offscreen.toBlob(blob => ...)` 生成 PNG，通过临时 `<a download>` 触发下载；
4. 导出不依赖任何外部服务，无 AI Token 时完整可用。

### 6. 图层重着色（Tint）与色彩系统

服装换色不预生成多色贴图，运行时重着色：

- 单层装备绘制到临时离屏 Canvas → `globalCompositeOperation = 'multiply'` 填充目标色 → 再用 `'destination-in'` 恢复原 alpha；
- 缓存每个 (装备ID, 色号) 组合的结果，避免每帧重算；
- 白色/浅灰底色的素材重着色效果最佳（这决定了素材选型时的底色偏好）；
- 该能力直接支撑舞台色彩系统的"服装配色方案实时预览"。

### 7. 性能预算

- 目标 60fps，单帧脚本耗时 ≤ 8ms；
- 精灵 `Image` 对象全局缓存池，禁止帧内 `new Image()`；
- 静止场景使用脏标记（dirty flag）跳过重绘；有动画/拖拽/同步事件时才全帧重绘；
- 40 个实体同屏（对应 30–40 人团体队形规模）为性能验收基线。

## 【v2 新增】开源人物精灵资源方案

**结论先行：原型期用 CC0 素材，商用产品（StageOS）只用 CC0 或自产素材，规避 SA/GPL 类授权。**

### 1. 候选资源对比（协议已核查）

| 资源 | 协议 | 商用 | 署名要求 | 改编同协议义务 | 适配度评估 |
| --- | --- | --- | --- | --- | --- |
| Kenney 全系素材 | CC0（公有领域） | ✅ | 无需署名 | 无 | 风格干净块状，含模块化角色包；适合原型与占位 |
| Dungeon Crawl Stone Soup tiles（32×32） | CC0 | ✅ | 无需署名 | 无 | 像素风角色/道具量大，风格偏暗黑地牢 |
| Quaternius 角色（3D 低模，含动画） | CC0 | ✅ | 无需署名 | 无 | 3D 低模可渲染转 2D 等距精灵，与低模拓扑工作流衔接 |
| Universal LPC Spritesheet 角色生成器 | GPL 3.0 + CC-BY-SA 3.0 双授权，逐文件混合 | ⚠️ 可但受限 | **必须署名全部作者**，随应用分发 CREDITS.csv 并在应用内可发现 | **SA 与 GPL 素材的改编件须同协议发布** | 部件化换装结构与槽位模型天然契合，但合规负担重 |
| OpenGameArt 综合站 | 逐素材混合（CC0/CC-BY/CC-BY-SA/OGA-BY/GPL） | 视素材 | 视素材 | 视素材 | 必须按协议筛选器过滤后逐件确认，禁止整站默认可用 |
| DawnLike（16×16） | CC-BY 4.0 | ✅ | 须署名 DragonDePlatino 与 DawnBringer | 无 | 尺寸偏小，适合复古风，不适合本项目视觉 |

### 2. 分阶段选型策略

- **原型期**：Kenney CC0 角色/道具 + 程序化几何图形兜底。零合规成本，跑通全部管线。
- **教学演示期**：可引入 LPC 生成器演示"部件化换装"的素材组织方式，但必须同步完成合规清单；课堂场景需明确告知学习者该素材不能直接进商用产品。
- **商用期（StageOS 接入）**：仅使用 CC0 素材或**自产素材**。自产管线 = AI 生成角色多视角参考图 → 低模拓扑 → 渲染输出 8 方向精灵帧（与 stageos-sprite-forge 技能对接），素材版权归自己，无协议风险。

### 3. LPC 合规清单（仅当确定使用 LPC 时执行）

1. 从生成器导出所选部件对应的 credits 文件，或直接分发整份 `CREDITS.csv`；
2. 在应用内提供用户可发现的署名入口（Credits 页面或可见链接）；
3. 标注为 CC-BY-SA / GPL 的部件，其改编版本（含改色、裁切后的衍生图）须以相同协议再发布——**这意味着改色后的服装图不能作为闭源商业素材**，商用场景直接排除此类部件，只保留其中 CC0 部件；
4. 素材包根目录放置 `LICENSE-assets.md`，逐文件记录协议与来源链接。

### 4. 精灵素材规格规范（无论来源，入库前必须满足）

- **方向**：等距视角至少 4 方向（前/后/左/右），推荐 8 方向；
- **帧**：idle 1–2 帧起步，walk 动画列为后续练习；
- **画布**：单帧统一尺寸（建议 128×192），透明背景 PNG；
- **锚点**：脚底中心，逐素材入库时校验；
- **命名**：`{slot}/{id}_{direction}_{frame}.png`，如 `top/vest-white_front_0.png`；
- **底色**：服装件优先白/浅灰底，保证运行时可重着色；
- **登记**：每个非 CC0 素材在 `CREDITS.csv` 登记名称、作者、协议、来源链接。

## Success criteria（v2 更新）

- 等距网格渲染：Canvas 正确显示菱形排列的等距 tile，坐标转换函数通过单元测试
- 精灵深度排序：角色在场景中移动时，与 tile 和其他角色的遮挡关系正确
- 【v2 新增】多层合成渲染：单角色 6 个槽位层按 L2a–L2f 固定序正确叠放，换装不产生像素级偏移（统一锚点校验通过）
- 【v2 新增】HiDPI 与导出：devicePixelRatio=2 下画面无模糊无偏移；离屏导出 2x PNG 成功且无外部依赖
- 拖拽换装：从装备面板拖拽装备到角色身上，角色外观即时更新，同一槽位替换旧装备
- 多人同步：两个浏览器窗口同时打开，一个窗口的换装操作在 500ms 内同步到另一个窗口
- 【v2 新增】素材合规：`CREDITS.csv` 覆盖全部非 CC0 素材，商用构建中不包含任何 SA/GPL 协议素材（CI 脚本校验）
- 自动化验证：`npm run validate` 覆盖所有步骤的核心检查点，GitHub Actions 工作流可通过

## Risks and open questions（v2 更新）

- ~~精灵资源缺失~~ → **已解决**：见「开源人物精灵资源方案」，原型期 Kenney CC0 + 程序化几何兜底，商用期 CC0 或自产素材。
- **素材协议传染性**：CC-BY-SA / GPL 素材的改编件（含改色衍生图）须同协议发布，闭源商用产品必须排除。CI 中增加素材协议扫描脚本，发现 SA/GPL 文件即失败。
- **等距坐标精度**：不同屏幕像素比（devicePixelRatio）下 Canvas 坐标计算可能出现偏移。已在渲染规范第 4 节预设 HiDPI 处理逻辑。
- **WebSocket 服务部署**：CI 中使用真实 WebSocket 服务，设置超时重试机制。
- **多人并发冲突**：采用"最后写入胜出"策略，乐观锁或操作队列留待后续练习。
- **拖拽兼容性**：本练习聚焦桌面端鼠标交互，移动端触控留待后续。
- **角色精灵格式**：使用独立 PNG 文件，精灵图集作为后续练习主题。
- 【v2 新增】**AI 自产精灵的一致性风险**：AI 生成多方向精灵帧时角色一致性依赖参考图锁定，8 方向帧间仍可能存在细节漂移；需人工校验锚点与轮廓，必要时人工修图后入库。
